RCJA Rescue Maze — Mighty Maisy 2026 Simulator

The robot sees the world through three VL53L1X-style ToF sensors and a TCS34725-style colour sensor, and is driven by a TB6612FNG-style signed-PWM motor pair. Same shape as your Arduino Mega build — same code structure, same noisy reality.

Mission

Identify every red and green victim in the maze. Avoid black "no-go" tiles. Return to the silver Start tile and exit. Score points: 10 per green, 25 per red, +25 for a clean exit, +25 each for accurate count bonuses (once you've found at least half the victims).

setup() and loop()

Define two functions. The simulator calls them like an Arduino sketch:

function setup() {
  // Runs ONCE
}
function loop() {
  // Runs ~50× per sec
}

Variables and helpers declared outside these functions are shared.

Drag these two hat blocks from the Setup category:

when program starts — runs once at the beginning
forever loop — runs ~50× per second

Variables set inside when program starts are visible in forever loop.

ToF distance sensors 
frontToF.read()  // mm or -1
leftToF.read()
rightToF.read()

Range up to ~1200 mm. Noisy: ±8 mm plus 1% of distance, occasional dropouts to -1. Always check for -1.

In the Sensors category:

front ▾ distance (mm)

The dropdown picks front, left, or right. Returns mm (or −1 on dropout).

Range up to ~1200 mm. Noisy: ±8 mm plus 1% of distance. Always compare with > 0 to filter dropouts.

Colour sensor 
const c = colourSensor.read();
// { r, g, b, c }  16-bit counts

Mounted 60 mm forward of the robot's centre (visible on the maze as the small dot with a dashed ring). This means the sensor sees colour before the robot's centre arrives — when red appears, you've usually got more driving to do before you're properly over the victim. Footprint ~20 mm radius. Write a classifier to turn raw RGB into a colour name.

⚠ Sim vs. real-hardware note. The simulator uses idealised colours — plain floor, silver, red, green, and black are well-separated and easy to distinguish. This is not the case on real hardware. Cardboard tile colour varies by batch, lighting changes between rooms, and the TCS34725's gain and integration time need calibration. When you port your code to the Mega + TCS34725, expect to recalibrate every threshold. A sample-and-tune-on-site routine is one of the first things you'll add to the real build.

In the Sensors category:

colour sensor red ▾ — reads one channel at a time

Dropdown picks: red, green, blue, or clear. Returns raw 16-bit counts.

Tip: Read clear to detect black (very low) or silver (very high). Compare red vs green to tell victims apart.

Mounted 60 mm forward of the robot's centre. The sensor sees colour before the robot arrives over the tile.

Encoders 
leftEncoder.read()    // ticks
rightEncoder.read()
leftEncoder.write(0)  // reset

2000 ticks per output-shaft revolution. Slip ±1% per tick — dead-reckoning drifts over distance.

In the Sensors category:

left ▾ encoder ticks — returns cumulative ticks
reset left ▾ encoder to 0 — resets the counter

2000 ticks per wheel revolution. Useful for measuring distance driven or turn angle.

Motors 
leftMotor.drive(speed)  // -255..+255
rightMotor.drive(speed)
leftMotor.brake()
leftMotor.standby()

Positive = forward. Top speed ~80 RPM (de-rated, matches real hardware). Crossing one tile takes ~1.2 s at full power.

In the Motion category:

left ▾ motor drive 150 — single motor, −255 to +255
drive both motors L 150 R 150 — set both at once
stop both motors

Positive = forward. To turn: make one side faster than the other. L > R = turn left.

Buttons 
buttonA.pressed()    // returns true once per press
buttonB.pressed()
buttonC.pressed()

Per RCJA rule 2.2.1, the real robot has manual start/restart controls. The sim exposes three buttons. You decide what each one does — strategy selection, encoder reset on LoP relocation, anything else.

Edge-triggered: pressed() returns true the first time you call it after a press fires, then false until the next press. Calling it more than once per tick will only see true once.

Presses can only fire at start (via the dropdown next to Run) or on Lack of Progress confirm (via the picker). Mid-run presses are not allowed by the rules.

In the Sensors category:

button A ▾ pressed? — true once per press

Dropdown picks A, B, or C. Use in an if block to react to operator button presses.

Presses fire at start (dropdown next to Run) or on Lack of Progress. Mid-run presses are not allowed.

Indicating victims and exit 
indicate('victim_unharmed')      // green tile
indicate('victim_harmed')        // red tile
indicate('exit')                 // Start tile, fully contained

// After half the victims found, on the Start tile,
// report your own tally (N). N must MATCH the count of
// valid indications you've made — false positives
// (indicating on empty floor) cause a mismatch and
// you lose the bonus.
indicate('count_unharmed', N)    // N = greens you've counted
indicate('count_harmed', N)      // N = reds you've counted

RCJA A6.2 requires stopping for ≥1 s over the victim. The sim doesn't enforce that — your code should.

The "Indications" row in the score panel shows the live count of valid indications (your indicate() calls that landed over a real victim, including duplicates). Match your reported N to those numbers.

In the Actions category:

indicate green victim (unharmed) ▾

Dropdown picks: green victim (unharmed), red victim (harmed), or exit.

report count of unharmed (green) ▾ as N

After finding ≥half the victims, return to Start and report your tallies. N must match your actual valid indications.

RCJA A6.2 requires stopping for ≥1 s over the victim. Use a wait block (1100 ms) after indicating.
